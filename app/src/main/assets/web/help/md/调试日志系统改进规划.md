# 调试日志系统改进规划
最好能把chapter对象、book对象等都记录下来，方便调试。
## 概述

调试日志系统是理解书源执行逻辑、排查问题的关键工具。当前系统已具备基础的日志记录能力，但要"真正看清一个源的执行逻辑和底层机制"，仍有改进空间。

本文档分析当前系统的能力边界，并提出改进方向和优先级，作为后续功能增加的参考依据。

---

## 一、当前系统能力

### 1. 架构概览

```
DebugEvent / FlowLogItem
    ├── requestId（请求ID，用于分组）
    ├── sourceUrl / sourceName（书源标识）
    ├── stage（流程阶段：NETWORK / PARSE / EXTRACT / REPLACE）
    ├── operation（操作类型：搜索/详情/目录/正文）
    ├── message（日志消息）
    ├── detail（详细信息）
    ├── rule / result / originalValue（规则执行相关）
    ├── url / method / statusCode / duration（网络请求相关）
    └── error（错误信息）
```

### 2. 核心组件

| 组件 | 文件位置 | 职责 |
|------|----------|------|
| FlowLogRecorder | `data/repository/debug/FlowLogRecorder.kt` | 流程日志记录器，异步记录各阶段日志 |
| DebugEvent | `model/debug/DebugEvent.kt` | 调试事件模型，统一采集调试信息 |
| DebugEventCenter | `data/repository/debug/DebugEventCenter.kt` | 调试事件中心，分发日志到各订阅者 |
| DebugLogScreen | `ui/debuglog/DebugLogScreen.kt` | 调试日志展示界面 |

### 3. 已实现的日志埋点

| 阶段 | 埋点位置 | 记录内容 |
|------|----------|----------|
| 网络请求 | `AnalyzeUrl.kt` | URL、method、statusCode、duration |
| 规则解析 | `AnalyzeRule.kt` | 规则内容、解析结果 |
| 字段提取 | `BookInfo.kt` / `BookList.kt` / `BookContent.kt` | 提取规则、提取结果 |
| 数据替换 | `AnalyzeRule.kt` / `BookContent.kt` | 替换规则、替换前后值 |

### 4. 当前系统能回答的问题

- ✅ 请求了什么 URL？用了什么方法？耗时多少？
- ✅ 执行了什么规则？得到了什么结果？
- ✅ 哪个阶段出了错？错误信息是什么？
- ✅ 整个流程耗时分布如何？

## 二、当前系统的局限

### 1. 规则执行的嵌套关系不清晰

**问题描述：**

一个复杂规则可能包含多层嵌套，例如：
```
@css:.content@textNodes##{{js表达式}}##正则
```

当前只记录"规则 → 最终结果"，但中间执行路径不可见：
- `@css:.content` 匹配到了哪些元素？
- `@textNodes` 如何处理这些元素？
- `{{js表达式}}` 执行时的变量环境是什么？
- 正则匹配的分组结果是什么？

**影响：**

无法理解复杂规则为什么得到当前结果，排查困难。

---

### 2. JS 执行环境状态缺失

**问题描述：**

JS 规则执行时，以下上下文信息对理解执行结果至关重要：
- `java` 对象的可用方法及调用结果
- `result` 变量的值及变化过程
- `src`、`book`、`chapter` 等变量的状态
- JS 执行过程中的函数调用链

当前只记录了 JS 执行的最终结果，中间状态不可见。

**影响：**

JS 规则调试困难，特别是涉及多步计算或条件判断时。

---

### 3. 网络请求细节不完整

**问题描述：**

当前网络日志记录：
- ✅ URL、method、statusCode、duration
- ❌ 完整请求头（包括自动添加的 header）
- ❌ 请求体（POST 数据）
- ❌ 响应头
- ❌ 响应体（至少前 N 个字符用于调试）
- ❌ 重定向链路
- ❌ Cookie 变化

**影响：**

无法精确复现网络请求，难以排查 header、cookie、POST 数据相关的问题。

---

### 4. 规则引擎内部决策不可见

**问题描述：**

规则引擎（AnalyzeRule）的内部逻辑是黑盒：
- XPath/JSONPath 匹配到了多少个节点？为什么选了这一个？
- 正则表达式的分组匹配结果
- `@get:{}` `@put:{}` 变量存取操作
- 规则类型判断逻辑（如何识别是 CSS/XPath/JSONPath/JS）

**影响：**

用户写了规则，但不知道系统是如何解析和执行的，学习成本高。

---

### 5. 数据流转全局视图缺失

**问题描述：**

搜索 → 详情 → 目录 → 正文，这个完整链路：
- `book` 对象在各阶段如何被填充？
- 阶段间的数据如何传递？
- 哪些字段是哪个阶段填充的？

当前日志分散在各阶段，缺少全局视图。

**影响：**

难以理解完整的数据流转过程，特别是字段来源追踪。

---

### 6. 错误上下文不足

**问题描述：**

当前错误记录：
- ✅ 错误类型和消息
- ❌ 错误发生前的几步操作
- ❌ 错误发生时的变量状态
- ❌ 完整堆栈（NoStackTraceException 故意省略）

**影响：**

错误信息不够，难以定位问题根因。

---

### 7. 并发和异步执行不透明

**问题描述：**

- 多个请求并发时，日志可能交错，难以追踪单一路径
- 协程的取消、超时处理逻辑不可见
- 预加载、缓存的命中情况不可见

**影响：**

性能问题和并发问题排查困难。

---

## 三、改进方向

### 方向一：网络请求完整详情

**目标：** 能够精确复现任何网络请求

**改进内容：**

| 信息 | 当前状态 | 改进后 |
|------|----------|--------|
| 请求 URL | ✅ 已记录 | ✅ |
| 请求方法 | ✅ 已记录 | ✅ |
| 请求头 | ❌ | ✅ 完整记录，包括自动添加的 header |
| 请求体 | ❌ | ✅ POST 数据（可截断） |
| 响应状态码 | ✅ 已记录 | ✅ |
| 响应头 | ❌ | ✅ 完整记录 |
| 响应体 | ❌ | ✅ 前 N 个字符（可配置） |
| 重定向链 | ❌ | ✅ 记录每次重定向 |
| Cookie 变化 | ❌ | ✅ 请求前后的 Cookie |

**实现要点：**

- 在 OkHttp 拦截器中捕获完整请求/响应信息
- 响应体需要缓存或截断，避免内存问题
- 敏感信息（如 token）需要脱敏选项

---

### 方向二：规则执行嵌套关系

**目标：** 看清规则执行的完整路径

**改进内容：**

```
规则: @css:.content@textNodes##{{js}}##regex

执行路径:
├── [1] @css:.content
│   ├── 输入: <html>...</html>
│   ├── 匹配: 3 个元素
│   └── 输出: <div class="content">...</div>
│
├── [2] @textNodes
│   ├── 输入: <div class="content">...</div>
│   └── 输出: "第一段\n第二段\n第三段"
│
├── [3] ##{{js}}
│   ├── 输入: "第一段\n第二段\n第三段"
│   ├── JS 环境: {result: "...", src: "..."}
│   └── 输出: "处理后内容"
│
└── [4] ##regex
    ├── 输入: "处理后内容"
    ├── 匹配分组: ["group1", "group2"]
    └── 输出: "最终结果"
```

**实现要点：**

- 在 AnalyzeRule 中增加执行路径追踪
- 每个子规则生成一个执行节点
- 支持折叠/展开查看详情

---

### 方向三：JS 执行环境状态

**目标：** 看清 JS 执行的上下文和过程

**改进内容：**

| 信息 | 说明 |
|------|------|
| 可用变量 | `result`、`src`、`book`、`chapter` 等的初始值 |
| 变量变化 | JS 执行过程中变量的赋值变化 |
| 函数调用 | `java.xxx()` 调用及返回值 |
| 执行耗时 | JS 执行时间 |
| 控制台输出 | `console.log` 输出 |

**实现要点：**

- 在 Rhino 执行环境中注入追踪逻辑
- 记录变量读写操作
- 可配置是否记录（避免性能影响）

---

### 方向四：错误上下文增强

**目标：** 错误发生时能快速定位根因

**改进内容：**

```
错误报告:
├── 错误信息: "正文为空"
├── 错误类型: ContentEmptyException
├── 发生位置: BookContent.kt:190
│
├── 前序操作:
│   ├── [1] 请求 https://xxx/chapter/1 (200, 1.2s)
│   ├── [2] 规则解析 @css:.content (匹配 0 个元素)
│   └── [3] 字段提取 content (结果为空)
│
├── 变量状态:
│   ├── url: "https://xxx/chapter/1"
│   ├── rule: "@css:.content"
│   └── source: "xxx书源"
│
└── 响应片段:
    └── "<html><body>...</body></html>" (前 500 字符)
```

**实现要点：**

- 维护最近 N 步操作的上下文
- 错误时自动关联上下文
- 提供响应片段用于分析

## 四、优先级排序

### P0 - 高优先级（核心排查能力）

| 方向 | 理由 |
|------|------|
| 网络请求完整详情 | 网络问题是书源问题的主要来源，需要精确复现 |
| 错误上下文增强 | 错误排查是最高频场景，当前信息不足 |

### P1 - 中优先级（规则理解能力）

| 方向 | 理由 |
|------|------|
| 规则执行嵌套关系 | 理解复杂规则执行的关键 |
| JS 执行环境状态 | JS 规则广泛使用，调试需求高 |

### P2 - 低优先级（深度理解能力）

| 方向 | 理由 |
|------|------|
| 数据流转全局视图 | 有助于理解整体流程，但非排查必需 |
| 规则引擎内部决策 | 学习成本高，对高级用户有价值 |
| 缓存和并发状态 | 性能优化场景使用 |

## 五、实现建议

### 1. 渐进式改进

- 不一次性改造所有模块
- 按 P0 → P1 → P2 顺序推进
- 每个方向完成后验证效果

### 2. 性能考虑

- 日志记录使用异步方式，不阻塞主流程
- 大数据（响应体）需要截断或采样
- 提供开关控制详细程度，避免影响正常使用性能

### 3. 存储策略

- 内存中保留最近 N 条日志（当前 500 条）
- 可选持久化到文件（用于问题复现）
- 支持导出日志用于分享和排查

### 4. 展示优化

- 提供多种视图：时间线、调用树、数据流
- 支持搜索和过滤
- 关键信息高亮显示

## 七、总结

当前调试日志系统已具备基础的"发生了什么"记录能力，但要真正理解"为什么这样执行"，需要补充：

1. **网络层**：完整的请求/响应详情
2. **规则层**：执行嵌套关系和中间结果
3. **JS层**：执行环境和变量状态
4. **错误层**：上下文和前序操作
5. **数据层**：全局流转视图
6. **引擎层**：内部决策过程

按优先级渐进式改进，最终目标是让调试日志能够完整还原书源的执行逻辑，成为书源开发和问题排查的利器。
