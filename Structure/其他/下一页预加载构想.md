## 方案5 架构设计

### 核心思路

将"一次性获取所有分页"改为"按需获取 + 预读取"，实现渐进式加载。

### 加载时序图

```
时间轴
  │
  │  用户打开章节
  ▼
  ├─► LazyContentManager.loadInitialPage()
  │        │
  │        ▼
  │    加载第 0 页（约 1-2 秒）
  │        │
  │        ▼
  │    返回 PageContent + nextUrl
  │        │
  │        ▼
  │    UI 显示第一页内容 ✓ 用户可以开始阅读
  │
  │  用户阅读中...（滚动到 70%）
  ▼
  ├─► LazyContentManager.prefetchNextPage() [后台]
  │        │
  │        ▼
  │    后台加载第 1 页（用户无感知）
  │        │
  │        ▼
  │    缓存到 pages[1]
  │
  │  用户继续滚动到底部
  ▼
  ├─► 检查 pages[1] 是否已缓存
  │        │
  │        ├─► 已缓存：直接显示 ✓
  │        │
  │        └─► 未缓存：显示加载中 → 等待加载完成 → 显示
  │
  │  用户继续滚动...
  ▼
  └─► 循环上述过程，直到无下一页
```

### 与现有架构的兼容性

| 场景 | 处理方式 |
|------|----------|
| 本地书籍 | 不启用懒加载（内容已在本地） |
| 单页正文 | 不启用懒加载（无下一页规则） |
| 多页正文 + 懒加载开启 | 启用懒加载模式 |
| 多页正文 + 懒加载关闭 | 使用传统一次性加载 |
| 缓存保存 | 合并所有已加载分页后保存 |

---

### 优势总结

| 优势 | 说明 |
|------|------|
| **初始加载快** | 只加载第一页，用户几乎无等待 |
| **服务器友好** | 请求分散到时间轴，不会瞬时大量请求 |
| **按需加载** | 用户可能不读完，后面的页面不需要加载 |
| **预加载无感** | 后台预加载，用户阅读时无感知 |
| **兼容现有架构** | 通过开关控制，不影响传统模式 |
| **失败容错** | 单页加载失败不影响已加载内容 |
